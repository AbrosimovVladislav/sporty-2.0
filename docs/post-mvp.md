# Пост-MVP: идеи и отложенные фичи

Список всего, что появляется в обсуждениях, но сознательно не попадает в MVP. Сюда складываем идеи без срочной реализации — чтобы они не терялись, но и не раздували скоуп.

Каждый пункт: что это, зачем, почему не в MVP.



ДЕОПоЗИТЫ

---

## Личные финансы игрока

**Что:** экран с личной историей расходов игрока: сколько сдал за событие, итого за период, задолженности.

**Зачем:** игроку удобно видеть свою сводку трат на спорт, сравнивать месяцы, понимать, сколько уходит.

**Почему не в MVP:** MVP фокусируется на финансах команды (сколько собрали, сколько ожидалось). Личный срез — надстройка над уже существующими данными `EventAttendance.paid`, её можно добавить позже без изменения схемы.

---

## Заявка «стать организатором» существующей команды

**Что:** игрок может подать заявку на получение роли организатора в команде, где он состоит как игрок. Существующие организаторы рассматривают заявку.

**Зачем:** путь для ситуации, когда организатор уходит или команда хочет распределить управление между несколькими людьми, не создавая новую команду.

**Почему не в MVP:** в MVP есть более простой путь — существующий организатор может промоутить игрока до со-организатора. Этого достаточно. Полноценный заявочный флоу (`JoinRequest` с типом `organizer-request`) — надстройка.

---

## Расширенная карточка игрока

**Что:** игровые характеристики (позиция, уровень, стиль игры, предпочтения), публичный профиль игрока, видимый другим.

**Зачем:** помогает в поиске игроков и команд, расширяет социальный слой.

**Почему не в MVP:** MVP — про команду и события, не про профили игроков. В MVP профиль — это минимум: имя, город, спорт.

---

## Поиск игроков

**Что:** организатор может искать свободных игроков по городу, виду спорта, доступности.

**Зачем:** добирать игроков в команду, когда не хватает на событие.

**Почему не в MVP:** сначала нужна наполненная база игроков. До этого — доходим органически через соцсвязи и приглашения.

---

## Поиск команд по району

**Что:** фильтр команд не только по городу, но и по району города.

**Зачем:** в больших городах игрок хочет найти команду рядом с домом.

**Почему не в MVP:** MVP закрывает базовый кейс «город + спорт». Район добавим после валидации базового поиска.

---

## Уведомления

**Что:** push / Telegram-уведомления: новое событие, изменение состава, подтверждение заявки, напоминание о голосовании.

**Зачем:** возврат пользователя в приложение без ручной проверки.

**Почему не в MVP:** требует отдельного инфра-пласта (Telegram Bot API для отправки сообщений, планировщик). MVP работает без них — пользователь сам открывает приложение.

---

## Множественные виды спорта

**Что:** поддержка не только футбола, но и других командных видов спорта.

**Зачем:** расширение аудитории.

**Почему не в MVP:** добавляет комплексность (разные правила событий, разные форматы составов). На старте — только футбол, чтобы быстрее валидировать ядро продукта.
