## ⬜ Итерация 53 — Финансы игрока + Достижения с реальной логикой

**Цель.** Закрыть две функциональные дыры в профиле: личные финансы игрока (важная для пользователя сводка) и реально работающие достижения вместо grayscale-плейсхолдеров.

### 53.1 Новый раздел «Финансы» в профиле игрока

**Проблема.** Сейчас в профиле нет персональных финансов. Игрок не видит свою сводку: сколько должен, сколько сдал, какие платежи прошли. Все данные есть в `attendances.paid` + `paid_amount`, но нигде не агрегированы для пользователя.

**Где разместить.** Варианты:

- **Вариант A (предпочтительный):** Добавить 5-й таб «Финансы» в профиле. 4 таба и так едва вмещаются — потребует уменьшить иконки или сократить названия
- **Вариант B:** Объединить «Результаты» и «Финансы» в один таб «Статистика» (статистика игр + финансы)
- **Вариант C:** Отдельная подстраница `/profile/finances` (доступна с обзорной строки в табе «Обо мне»)

**Решим при имплементации.** В наговорке пользователь склоняется к Варианту A («должна быть прям отдельная вкладка»).

**Содержимое раздела:**

- **Bignum баланс игрока** — сальдо: «Должен 4500 ₸» (если долг) или «Переплата 1200 ₸» (если переплатил)
- **2×2 stat:** «Сдал · N ₸ (за всё время)», «Должен сдать · M ₸», «Сыграно · K событий», «За месяц · L ₸»
- **График трат** (опц.) — bar по месяцам, как в финансах команды
- **История платежей** — список событий: дата, команда, сумма, статус (сдал / не сдал / переплата). Сортировка `start_at DESC`. Тап на строку → событие

### Backend

- `GET /api/users/[id]/finances` — новый endpoint. Агрегация по `attendances` где `attended=true`:
  - `totalPaid: sum(paid_amount)` за всё время
  - `totalExpected: sum(events.price_per_player)` где attended
  - `balance: totalPaid - totalExpected` (положительный = переплата, отрицательный = долг)
  - `monthlyPaid: sum(paid_amount)` за текущий месяц
  - `history: [{ event_id, date, team_name, expected, paid, status }]` — список последних 30 событий

### 53.2 Достижения: реальная логика и backend

**Проблема.** Сейчас в табе «Достижения» — grayscale-плейсхолдеры (Первый матч, 5 подряд, Капитан, MVP, 100% явка, 50 матчей). Никакая логика не работает, ачивки никогда не активируются.

**Решение.** Не вводить полноценную систему ачивок (это отдельная сущность с трекингом, post-MVP), а сделать минимальный набор computed-достижений, которые вычисляются on-the-fly из существующих данных:

- **«Первое событие»** — `attended_count >= 1`
- **«5 матчей подряд»** — посетил 5 событий своей команды подряд (без неприходов)
- **«Десятка»** — `attended_count >= 10`
- **«Полтинник»** — `attended_count >= 50`
- **«Организатор»** — есть membership с role = organizer хотя бы в одной команде
- **«Аккуратный»** — 0 неприходов из >= 5 событий

**Что убрать (нет данных для логики):** Капитан, MVP, 100% явка (без чёткого определения окна).

### Backend

- `GET /api/users/[id]/achievements` — возвращает массив `{ id, title, description, unlockedAt, icon }` с реально разблокированными ачивками; locked-плейсхолдеры тоже возвращаем для отображения «следующих целей»

### Frontend

- `AchievementCard` (`src/components/profile/AchievementCard.tsx`): unlocked — цветная, locked — grayscale
- В табе «Достижения» grid 3×N: разблокированные сверху, заблокированные с подзаголовком «Следующие цели» снизу

### Не входит

- Уведомления при разблокировке («Ты получил достижение!») — отдельная задача в нотификациях
- Ачивки на основе игровой статистики (голы, MVP) — нет трекинга

## ⬜ Итерация 54 — Публичный профиль игрока `/players/[id]` v2

**Цель.** Полный редизайн страницы публичного профиля. Сейчас она «выглядит ужасно» (формулировка пользователя), не следует дизайн-системе. Нужно превратить в красивую визитку игрока, с которой можно посмотреть на пользователя из поиска или приглашения.

**Зависит от.** Итерация 50.6 (`PositionChip`, закрыто в 1.5) — для отображения амплуа. Итерация 53.2 (`AchievementCard`) — для секции достижений.

### 54.1 Hero с фото

**Проблема.** Сейчас публичный профиль не имеет нормальной шапки — просто аватар поверх белого фона.

**Решение.** Использовать унифицированный hero-паттерн с фото:

- Hero 230px: фото пользователя (`avatar_url`) на полную ширину с blur-увеличением как фон + центральный круглый avatar 96px поверх
- Альтернатива: тёмная dark-карточка (как `EventHero`) с фото-фоном, скругление снизу 28px, имя + город · район поверх
- Back-arrow в top-left (тёмная таблетка над фото) — или нативный Telegram BackButton (см. 44.2 в 1.5)
- Если у игрока нет аватара — градиент по `teamGradient`-логике + крупный инициал

### 54.2 Метрики игрока

**Что показываем:**

- **Сыграно матчей** — bignum
- **Индекс надёжности** — % посещаемости (как в табе «Надёжность» личного профиля), с цветовым лейблом «Надёжный / Стабильный / Слабый»
- **Команд** — сколько команд, в которых состоит публично

Размещение: stat-row сразу под hero (как `HeaderStatGroup`, но в light-режиме на белом фоне).

### 54.3 Информация о игроке

- Био (если заполнено)
- Амплуа — `PositionChip` (см. 50.6 в 1.5)
- Уровень — `MiniBar` 1-5 (как в `PlayerListRow`)
- Бейдж «Ищет команду» (если включено)
- Возраст (если задана дата рождения)

### 54.4 Достижения

- Секция «Достижения» с теми же `AchievementCard` (см. 53.2). Если ачивок 0 — секцию не показываем
- Заголовок «Достижения · N» с эйбрау

### 54.5 Что НЕ показываем

- Список последних событий («где я был / где не был») — пользователь явно сказал, не нужно в публичном профиле
- Личные финансы — приватная информация
- Заявки в команды — тоже приватная

### 54.6 Будущий раздел «Отзывы»

Зарезервировать место (комментарий в коде или скрытая секция), но саму функциональность не реализуем — это post-MVP.

### Backend

- `GET /api/players/[id]` — расширить ответом: `attended_count`, `reliability_score`, `teams_count`, `achievements: []`, `is_looking_for_team`. Часть уже есть, проверим

## ⬜ Итерация 55 — Переработка онбординга

**Цель.** Привести онбординг (`/onboarding/`*) в визуальное и функциональное соответствие с остальным приложением и подготовить его к мульти-спорт-режиму. Сейчас это 5-шаговый wizard в старой вёрстке (name → birth-date → district → position → skill), документация по нему устарела.

**Зависит от.** Итерация 50.6 (`PositionChip`, закрыто в 1.5) — для шага выбора позиции. Итерация 44.1 (концепция города, закрыто в 1.5) — определяет, есть ли шаг «Город» и как он связан с глобальным scope.

### 55.1 Аудит и обновление документации

**Контекст.** Файл `docs/features/onboarding/[onboarding]-profile-setup.md` описывает single-screen с полями «имя / город / спорт» — это **устарело**. Реальная реализация — 5 шагов, других полей.

- Прочитать `src/app/onboarding/page.tsx` + все суб-роуты (`name`, `birth-date`, `district`, `position`, `skill`) + `onboarding-context.tsx` и зафиксировать актуальный flow
- Переписать `[onboarding]-profile-setup.md` под фактическую реализацию (после переработки в 55.2–55.5)

### 55.2 Welcome-экран `/onboarding`

**Проблема.** Сейчас при старте пользователь сразу попадает на первый шаг формы (имя). Нет ни приветствия, ни value prop, ни понимания «зачем эта форма и что меня ждёт».

**Решение.**

- Welcome-экран с логотипом приложения, slogan и 2-3 buletами «Что ты сможешь» (создавать команды / находить события / следить за финансами)
- Кнопка «Начать» → переход на первый шаг wizard'а
- Step-индикатор сверху (1/5, 2/5 и т.д.) на каждом шаге wizard'а — визуальная прогресс-полоса в зелёном `PageHeader`-стиле

### 55.3 Шаги wizard'а в дизайн-системе

**Проблема.** Все 5 шагов сейчас в старой вёрстке (до ит. 38 в 1.3). Цвета, типографика, кнопки — не из дизайн-системы.

**Решение.** Полный rewrite каждого шага на компонентах ДС:

- **Имя** — `Input` с дефолт-значением из Telegram + кнопка «Далее» в `BottomActionBar`
- **Дата рождения** — нативный date picker, нормализованный по ширине (тот же фикс, что в 48.1 в 1.5)
- **Район** — `Picker` / sheet-селектор (как в 52.6 в 1.5), не свободный ввод
- **Позиция** — multi-select `PositionChip` (см. 50.6 в 1.5), сразу показывает цветные чипы. Текст-подсказка «Можно выбрать несколько»
- **Уровень** — `SheetChipGroup` 1–5 с подписями («новичок» / «средне» / …)
- Каждый шаг: back-стрелка слева (на предыдущий шаг), title по центру, прогресс-индикатор сверху, primary-кнопка снизу в `BottomActionBar`

### 55.4 Шаг «Спорт» (новый)

**Контекст.** Сейчас `sport` хардкодится в `'football'` (см. устаревший доковый текст). После итерации 1.5.2 в БД будут команды и площадки сразу двух спортов (футбол + хоккей), значит пользователю надо выбрать на старте.

**Решение.**

- Добавить шаг «Спорт» (между «Имя» и «Дата рождения», или как первый шаг — определим при имплементации)
- Карточки выбора: «Футбол» / «Хоккей» с иконкой + одной строчкой описания
- Сохраняется в `users.sport`, дальше используется как фильтр scope в каталогах поиска и при создании команды
- **Важно.** Список позиций на шаге «Позиция» должен зависеть от выбранного спорта: футбол → ВРТ/ЗАЩ/ПЗЩ/НАП, хоккей → ВРТ/ЗАЩ/НАП. `PositionChip` (50.6 в 1.5) расширяется опциональным префиксом или просто получает sport-aware маппинг

### 55.5 Опциональные поля и quick-path

**Проблема.** Все 5 шагов сейчас обязательные. Импульсивный пользователь не может «быстро пройти и потом доделать».

**Решение.**

- Обязательные: имя, спорт, город (если решим оставить — см. 44.1 в 1.5)
- Опциональные: дата рождения, район, позиция, уровень — добавить кнопку «Пропустить» в `BottomActionBar` рядом с «Далее»
- На последнем шаге — кнопка «Готово» вместо «Далее»
- В пропущенных полях оставлять `null`; в профиле появится empty-state «Заполни профиль» (см. 52.3 в 1.5)

### 55.6 Финальный экран + первый landing

**Проблема.** Сейчас после `POST /api/onboarding` пользователь редиректится на `/home` без какого-либо приветствия или подсказки куда дальше.

**Решение.**

- После последнего шага — короткий success-экран «Готово, добро пожаловать!» с двумя CTA: «Найти команду» (→ `/search/teams`) / «Создать команду» (→ flow создания)
- Альтернатива: сразу на `/home`, но с onboarding-баннером сверху первой сессии «Что делать дальше?» (закрывается крестиком)
- **Решим при имплементации**
