## ✅ Итерация 44 — Концепция города и back-навигация (продуктовое решение)

**Цель.** Закрыть два архитектурных вопроса до старта работы по поиску и страницам 1-го уровня. Эта итерация — сначала продуктовое решение, потом реализация. Без согласования концепции дальнейшие итерации (50 «Поиск», 51 «Площадка», 52 «Профиль») могут потребовать переделки.

### 44.1 Концепция города (scope: глобально для всего приложения)

**Контекст.** Сейчас город задаётся в профиле пользователя (`users.city`) и используется частично — где-то фильтрует, где-то нет. Поиск событий/команд/игроков/площадок не имеет единого глобального scope «текущий город».

**Идея (Tinder-style):** город — глобальный scope, который скоупит всё приложение (события, команды, игроки, площадки). Меняется в одном-двух местах:

- На главной странице — кликабельная плашка «Алматы» (или текущий город) → bottom-sheet выбора города
- В профиле — раздел «Локация» с тем же селектором
- Изменение города **не меняет постоянный city** в профиле; это **просмотровый scope** (как в Тиндере «search location»)

**Эффект.**

- Все каталоги поиска — фильтруются по выбранному городу автоматически (без отдельного фильтра «Город» в `FiltersSheet`)
- Главная — показывает события/команды только этого города
- Видимость в обратную сторону: пользователь из Астаны попадает в выдачу Алматы, только если активный scope — «Алматы»

**Что решить:**

1. Принимаем эту модель или альтернативу (мульти-город — показывать все, фильтровать опционально)?
2. Хранить выбранный scope-город в localStorage / cookie / в БД (`users.preview_city`)?
3. Что показываем при первом запуске — город профиля по умолчанию?
4. Где визуально отображается текущий scope, кроме главной?

### 44.2 Back-навигация со страниц 1-го уровня

**Контекст.** Когда пользователь идёт `/search/teams` → тап на команду → `/team/[id]`, он попадает на страницу первого уровня (то же, что Моя Команда с предвыбранной). Из неё нет очевидного способа вернуться обратно на поиск.

**Та же проблема для:**

- `/search/players` → `/players/[id]` (профиль)
- `/search/venues` → `/venues/[id]` (создаётся в 51.1) — будет back-arrow в hero
- `/search/events` → `/team/[id]/events/[eventId]` — back уже есть (event hero)

**Гипотеза.** Использовать **Telegram BackButton** (нативный, `@tma.js/sdk-react`), не загромождая UI:

- На страницах 1-го уровня (главная, поиск, профиль, моя команда) — BackButton скрыт
- На страницах 2-го уровня (внутри команды/события/игрока/площадки) — BackButton показан и ведёт назад через `router.back()`
- Свайп-назад на iOS/Android Telegram работает нативно

**Что решить:**

1. Использовать только BackButton, или дополнительно рендерить back-arrow в `PageHeader` слева для непривычных пользователей?
2. На мобильном вебе (без Telegram-клиента) — fallback на собственную back-кнопку или нативный browser back?

**Не входит.** Полный аудит всех роутов на наличие back — делаем точечно по мере прохождения итераций 51–54.

## ⬜ Итерация 46 — Единая карточка игрока в команде (`TeamPlayerSheet`)

**Цель.** Унифицировать карточку игрока внутри команды как один паттерн со всех экранов раздела.

**Проблема.** Сейчас тап на игрока в составе открывает `PlayerSheet` (с promote/remove для орга), но из других мест (лидеры на главной команды, должники на финансах) тап ведёт на `/players/[id]` (полноценный публичный профиль). Нет единого паттерна. По правилу «внутри Моей команды все тапы по игрокам открывают карточку игрока в команде» нужен один компонент со всех точек входа.

### 46.1 Компонент `TeamPlayerSheet`

**Содержимое (всегда видимая шапка):**

- Avatar 64+px, имя, бейдж «Организатор» (если орг)
- Амплуа (позиции — все, что у игрока), уровень, город
- Кнопка «Открыть профиль» → `/players/[id]` (публичный профиль)

**Аккордеоны** (раскрываются по тапу):

1. **Надёжность · в этой команде** — посещаемость (%), неприходы, отмены, последние события с цветными точками — то же, что в `/profile` → таб «Надёжность», но скоупленное на эту команду
2. **Финансы · в этой команде** — суммарно: должен был сдать, сдал, сальдо (долг/переплата); история платежей по событиям (был / сдал); может «открыть карточку события» из строки
3. **Игровая статистика** — placeholder под будущее (голы / передачи / жёлтые / MVP); сейчас «скоро»

**Действия для организатора** (внизу sheet'а): «Сделать организатором», «Удалить из команды» — как сейчас в `PlayerSheet`. Для самого пользователя: «Покинуть команду» (как сейчас, через подтверждение).

**Где открывается:**

- `/team/[id]/roster` — тап на строку игрока
- `/team/[id]` (главная) → «Лидеры 30 дней» — тап на лидера (вместо текущего перехода в публичный профиль)
- `/team/[id]/finances` → «Должны» / «Переплатили» — тап на строку (унифицируем в итерации 49)
- Везде, где появится список игроков команды в будущем

### 46.2 Главная команды: «Лидеры 30 дней» — уточнить смысл

**Проблема.** Карточка «Лидеры 30 дней» — непонятно, лидеры чего (посещений? платежей? игр?).

**Решение.** Подзаголовок/уточнение: «Лидеры по посещаемости · 30 дней» (если сейчас агрегат именно по посещаемости — назвать прямо). Тап на лидера → `TeamPlayerSheet` (46.1).

### Не входит

- Реальная игровая статистика — нет трекинга, остаётся placeholder-аккордеон
- Замена `PlayerSheet` на `TeamPlayerSheet` в `/search/players` — там остаётся переход на публичный профиль (другой контекст)

### 47.2 Баг профиля: множественные позиции сохраняются криво

**Проблема.** В `/profile/settings`:

- Выбираю только «Полузащитник», сохраняю → отображается «Полузащитник + Нападающий»
- Не сохраняется именно тот набор, который выбрал пользователь

**Решение.** Найти `PUT /api/users/[id]/profile` и логику update в `profile/settings/page.tsx`, починить — позиции записываются строго тем массивом, что в форме (replace, не merge).

**Принципиально.** Один игрок может иметь несколько амплуа (это нормально и должно поддерживаться), но это должен быть осознанный выбор пользователя, а не побочка кривой логики.

### 47.3 Состав: фильтр по позициям должен учитывать ВСЕ амплуа игрока

**Проблема.** Если у игрока две позиции (`Полузащитник`, `Нападающий`):

- В фильтре `Полузащитник` он есть
- В фильтре `Нападающий` его нет (или наоборот, в зависимости от того, какая позиция «первая»)
- В фильтре `Все` — есть

**Должно быть.** Игрок попадает в выдачу любого фильтра, который пересекается с его массивом позиций. Multi-position игрок видим во всех своих чипах.

**Решение.** Проверить логику фильтрации на `/team/[id]/roster`. Поменять `position === selectedPosition` на `positions.includes(selectedPosition)` или эквивалент. Тот же фикс на `/search/players` — в итерации 50 (пункт 50.4).

### 47.4 События команды: исправить сортировку

**Проблема.** Сейчас в `/team/[id]/events`:

- Предстоящие: 1 мая, 30 апреля, 29 апреля — далёкое сверху, ближайшее снизу. Должно быть наоборот
- Прошедшие — нужно проверить, что самое свежее сверху

**Решение.**

- Секция «Предстоящие» — sort `start_at ASC` (ближайшее сверху)
- Секция «Прошедшие» — sort `start_at DESC` (последнее завершённое сверху)

### 49.1 Hero-KPI и сводка: понятные лейблы и корректный trend

**Проблемы:**

- «Лучше прошлого месяца» при `realBalance = -8600` и прошлый = 0 — сравнение явно сломано или формулировка вводит в заблуждение
- «Касса» — название непрозрачное (что это? касса чего?)
- «Долги игроков» — формулировка ок по смыслу, но визуально не выделена как «то, что должны нам»
- «К оплате» — непонятно: кому платим? кто платит?
- Сводка из 4-х показателей (собрано / расходы / переплаты / ожидаемый сбор) — набор без приоритизации, всё одного веса

**Решение:**

- Hero-карточку и сводку оставить визуально похожими, но переписать лейблы и переработать иерархию
- Лейблы должны читаться без контекста: «Должны нам · игроки», «Должны мы · площадкам», «На счету · депозит» (или похоже)
- Trend-сравнение (`+12%`, «Лучше / хуже прошлого месяца») — починить логику или убрать, если для текущего объёма данных оно вводит в заблуждение
- Из 4-х метрик сводки — оставить 2-3 действительно важных (определим при имплементации)

### 49.2 График «Поток за 6 месяцев» — добавить читаемость

**Проблема.** Бар-график есть, но на нём нет цифр, непонятно, что мы видим и что сравнивается. Тап неактивен (и не должен — пока).

**Решение:**

- Подписать значения на столбиках (или хотя бы максимум / итог)
- Чёткий лейбл осей (месяцы снизу, валюта/единица сверху или в подписи)
- Легенда «Сборы / Расходы» вместо `collected` / `venuePaid`
- Показывать «Чистый: +N ₸» под графиком (уже планировалось в ит. 42, проверить рендер)

### 49.3 «Эффективность сборов» → «Маржинальность»

**Проблема.** Сейчас donut показывает «181%» — это `collected / expected`. При expected = 0 или маленьком оно выходит за 100% и теряет смысл. Сама метафора donut «эффективность 87%» в текущем виде — это про то, насколько игроки сдали свою долю. Это полезно, но более содержательная метрика — **маржинальность** (доходы vs расходы команды).

**Решение:**

- Поменять смысл donut'а: показать маржинальность = `(collected - venuePaid) / collected` или похожую формулу (определим при имплементации)
- Подзаголовок: «Доходы N ₸ · Расходы M ₸»
- Цветовая логика: зелёный, если маржа положительная; красный, если отрицательная
- Альтернатива: не donut, а разделённая на два сегмента полоса (доходы/расходы) с числами — финальный выбор при имплементации

**Проблема.** Сейчас в профиле нет персональных финансов. Игрок не видит свою сводку: сколько должен, сколько сдал, какие платежи прошли. Все данные есть в `attendances.paid` + `paid_amount`, но нигде не агрегированы для пользователя.

**Где разместить.** Варианты:

- **Вариант A (предпочтительный):** Добавить 5-й таб «Финансы» в профиле. 4 таба и так едва вмещаются — потребует уменьшить иконки или сократить названия
- **Вариант B:** Объединить «Результаты» и «Финансы» в один таб «Статистика» (статистика игр + финансы)
- **Вариант C:** Отдельная подстраница `/profile/finances` (доступна с обзорной строки в табе «Обо мне»)

**Решим при имплементации.** В наговорке пользователь склоняется к Варианту A («должна быть прям отдельная вкладка»).

**Содержимое раздела:**

- **Bignum баланс игрока** — сальдо: «Должен 4500 ₸» (если долг) или «Переплата 1200 ₸» (если переплатил)
- **2×2 stat:** «Сдал · N ₸ (за всё время)», «Должен сдать · M ₸», «Сыграно · K событий», «За месяц · L ₸»
- **График трат** (опц.) — bar по месяцам, как в финансах команды
- **История платежей** — список событий: дата, команда, сумма, статус (сдал / не сдал / переплата). Сортировка `start_at DESC`. Тап на строку → событие

### Backend

- `GET /api/users/[id]/finances` — новый endpoint. Агрегация по `attendances` где `attended=true`:
  - `totalPaid: sum(paid_amount)` за всё время
  - `totalExpected: sum(events.price_per_player)` где attended
  - `balance: totalPaid - totalExpected` (положительный = переплата, отрицательный = долг)
  - `monthlyPaid: sum(paid_amount)` за текущий месяц
  - `history: [{ event_id, date, team_name, expected, paid, status }]` — список последних 30 событий

### 53.2 Достижения: реальная логика и backend

**Проблема.** Сейчас в табе «Достижения» — grayscale-плейсхолдеры (Первый матч, 5 подряд, Капитан, MVP, 100% явка, 50 матчей). Никакая логика не работает, ачивки никогда не активируются.

**Решение.** Не вводить полноценную систему ачивок (это отдельная сущность с трекингом, post-MVP), а сделать минимальный набор computed-достижений, которые вычисляются on-the-fly из существующих данных:

- **«Первое событие»** — `attended_count >= 1`
- **«5 матчей подряд»** — посетил 5 событий своей команды подряд (без неприходов)
- **«Десятка»** — `attended_count >= 10`
- **«Полтинник»** — `attended_count >= 50`
- **«Организатор»** — есть membership с role = organizer хотя бы в одной команде
- **«Аккуратный»** — 0 неприходов из >= 5 событий

**Что убрать (нет данных для логики):** Капитан, MVP, 100% явка (без чёткого определения окна).

### Backend

- `GET /api/users/[id]/achievements` — возвращает массив `{ id, title, description, unlockedAt, icon }` с реально разблокированными ачивками; locked-плейсхолдеры тоже возвращаем для отображения «следующих целей»

### Frontend

- `AchievementCard` (`src/components/profile/AchievementCard.tsx`): unlocked — цветная, locked — grayscale
- В табе «Достижения» grid 3×N: разблокированные сверху, заблокированные с подзаголовком «Следующие цели» снизу

### Не входит

- Уведомления при разблокировке («Ты получил достижение!») — отдельная задача в нотификациях
- Ачивки на основе игровой статистики (голы, MVP) — нет трекинга

**Явки и платежи:**

- Для каждого прошедшего события: ~70% игроков команды отметили «приду», из них ~80% реально были (`attended=true`)
- Платежи: ~70% присутствовавших оплатили в полном объёме, ~20% частично, ~10% не платили — даёт реалистичные сценарии «должников» и «переплат» в финансах команды

- Несколько игроков с переплатой

**Заявки:** 2-3 pending-заявки в команду A (для теста red-dot на gear в 45.1).

### 56.6 npm-скрипты

```
"db:reset": "supabase db reset --linked"           # дроп всех таблиц + миграции с нуля
"db:seed": "node scripts/seed.js"                   # запускает seed.sql через psql/Supabase REST
"db:fresh": "npm run db:reset && npm run db:seed"   # full reset + seed одной командой
```

**Имплементация.**

- Миграции применяем через Supabase MCP (`apply_migration`), как зафиксировано в memory
- `scripts/seed.js` — простой скрипт, читает SQL-файлы из `supabase/seed/` и применяет через service-role клиент (`SUPABASE_SERVICE_ROLE_KEY`)
- Документация в `docs/tech/db-seed.md` — как пользоваться, какие данные внутри, как добавлять новые

### Не входит

- Seed для футбольных данных — только хоккей (по запросу пользователя). Если позже понадобится — отдельная задача
- Тестовые сценарии типа «Telegram-юзер X = organizer team A» — пользователь подключается к seed-данным как новый пользователь через свой Telegram-аккаунт
