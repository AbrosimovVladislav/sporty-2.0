## ⬜ Итерация 53 — Финансы игрока + Достижения с реальной логикой

**Цель.** Закрыть две функциональные дыры в профиле: личные финансы игрока (важная для пользователя сводка) и реально работающие достижения вместо grayscale-плейсхолдеров.

### 53.1 Новый раздел «Финансы» в профиле игрока

**Проблема.** Сейчас в профиле нет персональных финансов. Игрок не видит свою сводку: сколько должен, сколько сдал, какие платежи прошли. Все данные есть в `attendances.paid` + `paid_amount`, но нигде не агрегированы для пользователя.

**Где разместить.** Варианты:

- **Вариант A (предпочтительный):** Добавить 5-й таб «Финансы» в профиле. 4 таба и так едва вмещаются — потребует уменьшить иконки или сократить названия
- **Вариант B:** Объединить «Результаты» и «Финансы» в один таб «Статистика» (статистика игр + финансы)
- **Вариант C:** Отдельная подстраница `/profile/finances` (доступна с обзорной строки в табе «Обо мне»)

**Решим при имплементации.** В наговорке пользователь склоняется к Варианту A («должна быть прям отдельная вкладка»).

**Содержимое раздела:**

- **Bignum баланс игрока** — сальдо: «Должен 4500 ₸» (если долг) или «Переплата 1200 ₸» (если переплатил)
- **2×2 stat:** «Сдал · N ₸ (за всё время)», «Должен сдать · M ₸», «Сыграно · K событий», «За месяц · L ₸»
- **График трат** (опц.) — bar по месяцам, как в финансах команды
- **История платежей** — список событий: дата, команда, сумма, статус (сдал / не сдал / переплата). Сортировка `start_at DESC`. Тап на строку → событие

### Backend

- `GET /api/users/[id]/finances` — новый endpoint. Агрегация по `attendances` где `attended=true`:
  - `totalPaid: sum(paid_amount)` за всё время
  - `totalExpected: sum(events.price_per_player)` где attended
  - `balance: totalPaid - totalExpected` (положительный = переплата, отрицательный = долг)
  - `monthlyPaid: sum(paid_amount)` за текущий месяц
  - `history: [{ event_id, date, team_name, expected, paid, status }]` — список последних 30 событий

### 53.2 Достижения: реальная логика и backend

**Проблема.** Сейчас в табе «Достижения» — grayscale-плейсхолдеры (Первый матч, 5 подряд, Капитан, MVP, 100% явка, 50 матчей). Никакая логика не работает, ачивки никогда не активируются.

**Решение.** Не вводить полноценную систему ачивок (это отдельная сущность с трекингом, post-MVP), а сделать минимальный набор computed-достижений, которые вычисляются on-the-fly из существующих данных:

- **«Первое событие»** — `attended_count >= 1`
- **«5 матчей подряд»** — посетил 5 событий своей команды подряд (без неприходов)
- **«Десятка»** — `attended_count >= 10`
- **«Полтинник»** — `attended_count >= 50`
- **«Организатор»** — есть membership с role = organizer хотя бы в одной команде
- **«Аккуратный»** — 0 неприходов из >= 5 событий

**Что убрать (нет данных для логики):** Капитан, MVP, 100% явка (без чёткого определения окна).

### Backend

- `GET /api/users/[id]/achievements` — возвращает массив `{ id, title, description, unlockedAt, icon }` с реально разблокированными ачивками; locked-плейсхолдеры тоже возвращаем для отображения «следующих целей»

### Frontend

- `AchievementCard` (`src/components/profile/AchievementCard.tsx`): unlocked — цветная, locked — grayscale
- В табе «Достижения» grid 3×N: разблокированные сверху, заблокированные с подзаголовком «Следующие цели» снизу

### Не входит

- Уведомления при разблокировке («Ты получил достижение!») — отдельная задача в нотификациях
- Ачивки на основе игровой статистики (голы, MVP) — нет трекинга

**Явки и платежи:**

- Для каждого прошедшего события: ~70% игроков команды отметили «приду», из них ~80% реально были (`attended=true`)
- Платежи: ~70% присутствовавших оплатили в полном объёме, ~20% частично, ~10% не платили — даёт реалистичные сценарии «должников» и «переплат» в финансах команды

- Несколько игроков с переплатой

**Заявки:** 2-3 pending-заявки в команду A (для теста red-dot на gear в 45.1).

### 56.6 npm-скрипты

```
"db:reset": "supabase db reset --linked"           # дроп всех таблиц + миграции с нуля
"db:seed": "node scripts/seed.js"                   # запускает seed.sql через psql/Supabase REST
"db:fresh": "npm run db:reset && npm run db:seed"   # full reset + seed одной командой
```

**Имплементация.**

- Миграции применяем через Supabase MCP (`apply_migration`), как зафиксировано в memory
- `scripts/seed.js` — простой скрипт, читает SQL-файлы из `supabase/seed/` и применяет через service-role клиент (`SUPABASE_SERVICE_ROLE_KEY`)
- Документация в `docs/tech/db-seed.md` — как пользоваться, какие данные внутри, как добавлять новые

- Seed для футбольных данных — только хоккей (по запросу пользователя). Если позже понадобится — отдельная задача
- Тестовые сценарии типа «Telegram-юзер X = organizer team A» — пользователь подключается к seed-данным как новый пользователь через свой Telegram-аккаунт

### 55.2 Производительность: первая загрузка

Направления (приоритизация после измерений):

- Bundle size: проверить `npm run build` output, найти тяжёлые зависимости, посмотреть на dynamic imports для редко используемых компонентов (sheet'ы, графики)
- Supabase-запросы: дублирующиеся запросы при навигации между табами одной команды/профиля — кэшировать через context или React Query (если введём)
- Изображения: проверить, что `avatar_url`, `logo_url`, `photo_url` отдаются через Next.js `<Image>` с правильными `sizes`/`priority`
- Lazy-load для второстепенных секций (графики финансов, список достижений)
- Telegram Mini App init: проверить, не блокируем ли первый рендер ожиданием SDK

### 55.3 Производительность: in-app навигация

- Проверить, что переходы между табами `/team/[id]/*` не перезапрашивают одинаковые данные
- `team-context` — посмотреть, не происходит ли refetch при каждом монтировании
- Skeleton-стейты: убедиться, что они есть на всех страницах, где данные грузятся > 200ms
