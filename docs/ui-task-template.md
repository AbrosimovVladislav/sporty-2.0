# Шаблон UI-задачи

Этот документ — обязательное чтение перед началом любой задачи по интерфейсу. Если ты модель (Codex, Cursor, Claude и т.д.), читай **до** того, как написать первую строку кода.

---

## Перед началом

Прочитай:
1. [design/design-system.md](design/design-system.md) — целиком. Не «по диагонали».
2. [ui-consistency-audit.md](ui-consistency-audit.md) — найди свой пункт. Если задача не в списке — это сигнал, что её нужно сначала описать.
3. Документ соответствующего эпика в [features/](features/) — чтобы понять текущую бизнес-логику экрана.

---

## Запрещено

- **Не вводи новую визуальную систему.** Все цвета, радиусы, отступы, тени — только из дизайн-системы.
- **Не используй хардкод-цвета** Tailwind типа `text-green-600`, `bg-red-500`, `border-gray-300`. Только токены: `text-primary`, `bg-danger`, `border-border`.
- **Не используй `font-display` (Oswald)** на новых/переписываемых экранах. Только `font-sans` + жирность.
- **Не используй `bg-background-dark`** на не-фото-блоках. Тёмный фон допустим только под фото с градиентом (см. `PhotoBanner`).
- **Не добавляй тяжёлые тени** (`shadow-xl`, `shadow-2xl`). Только `shadow-card` и `shadow-pop`.
- **Не лепи `border` вокруг каждой карточки** — по умолчанию карточка `bg-background-card rounded-lg p-4 shadow-card`, без рамки.
- **Не добавляй UPPERCASE-заголовки** заголовков карточек, кнопок, бейджей. Uppercase допустим только в `SectionEyebrow` (мелкий эйбрау-текст).
- **Не делай новых компонентов** без явной задачи. Если нужен новый паттерн — сначала **опиши его в дизайн-системе** в виде PR/предложения, потом реализуй.
- **Не дублируй данные.** Если число уже в эйбрау — не повторяй его справа от блока. Если есть «+12» — не пиши «всего 16» рядом.
- **Не оставляй текст «Сохраняю…», «Создаю…»** в кнопке. Используй `Button loading` со спиннером.
- **Не используй юникод-эмодзи** (📍 ✓ ×) в постоянном UI — только SVG из `Icons.tsx`.
- **Не правь `globals.css`** без обновления списка токенов в `design/design-system.md`.
- **Не лезь в чужой пункт аудита.** Делай только то, что в задаче.

---

## Обязательно

- Используй компоненты из `src/components/ui/` (`Card`, `Button`, `Pill`, `Avatar`, `AvatarStack`, `MiniBar`, `SectionEyebrow`, `IconButton`, `BottomActionBar`, `EmptyState`, `ScreenHeader`, `PhotoBanner`, `StatCard`).
- Сверяй каждое нетривиальное место с разделом «Компоненты» в `design/design-system.md`.
- Числа выводи с `tabular-nums`, чтобы списки не «прыгали».
- Скрывай `BottomTabs` через `hideBottomTabs` контекст, если экран имеет `BottomActionBar`.
- Тап-таргеты — минимум 44px.
- Состояния: загрузка (`Skeleton`), пустота (`EmptyState`), ошибка (карточка с текстом `text-danger`).
- После изменений — обнови соответствующий документ в `features/` (это правило из CLAUDE.md).

## Шаблон задачи

При нарезке задач для модели/исполнителя используй этот формат. Каждый пункт — самостоятельная задача, которую можно отдать одной модели за один заход.

```
## Задача: <короткое название, 5–8 слов>

### Цель
Одна-две фразы что должно получиться. С точки зрения пользователя.

### Контекст
- Пункт аудита: <ui-consistency-audit.md → раздел N.M>
- Итерация роадмапа: <24/25/26/27/28/29>
- Связанные документы: <features/.../*.md>

### Файлы
Какие файлы будут затронуты. Если задача создаёт новые файлы — указать их пути.
- src/...
- src/...

### Что прочитать перед работой
- docs/design/design-system.md — целиком (обязательно)
- docs/ui-consistency-audit.md — пункт N.M
- <feature-doc>.md — для бизнес-логики

### Что изменить
Конкретные действия списком. Должно быть однозначно интерпретируемо.
1. ...
2. ...
3. ...

### Что не трогать
- Файлы / зоны / поведение, которые НЕ должны измениться в рамках этой задачи.
- Типичные «не трогать»:
  - бизнес-логика API
  - другие экраны
  - схема БД
  - токены globals.css (если задача не в итерации 24)

### Критерий приёмки
Чек-лист:
- [ ] Визуально совпадает с правилами дизайн-системы (раздел «...»)
- [ ] Нет `font-display`, `bg-background-dark`, хардкод-цветов
- [ ] Состояния загрузка/пустота/ошибка покрыты
- [ ] `tabular-nums` на числах
- [ ] Тап-таргеты ≥ 44px
- [ ] Документация обновлена (если применимо)
- [ ] `npm run lint` проходит
- [ ] `npm run build` проходит

### Ручная проверка
Шаги для запуска и проверки в браузере:
1. `npm run dev`
2. Открыть `<URL>`
3. Проверить:
   - <сценарий 1>
   - <сценарий 2>
   - <узкий экран 320px>
```

## Примеры коротких задач (правильно нарезано)

### Хорошо ✅

> **Задача: Заменить тёмный hero на светлый ScreenHeader на /home**
> Цель: на главной убрать тёмный блок приветствия, использовать светлый заголовок «Привет, Имя» и подпись городом.
> Файлы: `src/app/(app)/home/page.tsx`.
> Не трогать: блок ближайшего события, быстрые действия, секцию «Мои команды» — это отдельные задачи.

### Плохо ❌

> **Задача: Переписать главную**
> Слишком крупно. Модель не сможет удержать всё в голове, начнёт фантазировать. Разбей на 4–5 микрозадач (header, карточка события, действия, мои команды, empty state).

## Если в процессе становится понятно, что нужен новый компонент

1. **Остановись.** Не пиши его inline.
2. Опиши в `design/design-system.md` (раздел «Компоненты»):
   - название,
   - HTML-структура,
   - токены, которые он использует,
   - где будет применяться.
3. Заведи отдельную задачу в итерацию 24 (или текущую) на его создание в `src/components/ui/`.
4. Только после этого продолжай свою основную задачу.

Это правило защищает от того, что в проекте появятся 5 разных вариантов «карточки» или «кнопки».

---

## Если требование задачи противоречит дизайн-системе

Не «адаптируй на лету». Останови работу, задай вопрос человеку:
- какое из правил отменяется,
- временно или навсегда,
- нужно ли обновить `design/design-system.md`.

Дизайн-система — источник правды. Изменения в ней — отдельный коммит, потом задача.
